HOST_A: Good. Paper two — DTF, from OpenKedge.
HOST_B: This one is a formal architecture paper, no benchmarks. The core idea, in plain English: today, when an agent does something dangerous, you check its identity and its permissions — does this agent have the right scope. DTF says that's backwards. You should be checking the justification for this specific action, in this specific context, right now. And the authority to execute should be derived from that justification, not from ambient credentials.
HOST_A: Walk me through that. What does "derived from the justification" mean operationally?
HOST_B: So before any high-stakes mutation, the agent has to produce a Justification Proof — a structured object that binds together the intent, the context snapshot, and the policy bundle. Then independent evaluators — multiple of them — have to attest to that proof. Only after consensus does the system mint an ephemeral execution identity, scoped exactly to what the proof authorized.
HOST_A: So it's not "this agent is allowed to send emails" — it's "this agent, with this proof, on this evidence, is allowed to send this email."
HOST_B: Exactly. And everything — proof, attestations, decision, execution, outcome — gets appended to an Evidence Chain whether the action was approved, rejected, or escalated. The whole sequence is replayable.
HOST_A: That's basically the audit story regulated tenants keep asking us for.
HOST_B: It is. The honest objections — first, it's a formal model implemented on the authors' own substrate, no independent evaluation, no adversarial test. Second, multi-evaluator consensus on every high-stakes mutation has real latency cost; you can't do this on every tool call. Third, and this is the one I keep getting stuck on — how do you define "high-stakes" in a way an adversarial agent can't game by decomposing a dangerous action into a sequence of individually trivial steps? The paper doesn't really answer that.
HOST_A: That last one feels like the actual hard problem.

HOST_A: That's a scoping exercise we can do this quarter regardless. Okay, paper three — SkillSmith.
HOST_B: This one I'm the most excited about because it's immediately operational. Plain English: when you give an agent a tool, you usually shove a bunch of prose context into the prompt explaining what the tool does, when to use it, edge cases, examples. SkillSmith looked at this and found that about 51% of those tokens are irrelevant to the actual execution path. And worse — agents keep reconstructing nearly the same reasoning trace for the same skill across different tasks. Almost half the reasoning is redundant.
HOST_A: So they're saying we're paying for the same thinking, over and over.
HOST_B: Right. Their fix is compiler-style. Offline, they analyze each skill package and classify its shape — is it procedural steps, is it a callable script, is it reference documentation, or is it ambiguous. Then they lower it into a minimal internal representation, and expose a Boundary Contract — a clean ABI with typed inputs and outputs, risk metadata, action policies, fallback obligations. The agent at runtime only sees the contract, not the prose.
HOST_A: ABI is — application binary interface, right? Like the published surface of a compiled library?
HOST_B: Exactly that, but for agent skills. And they use progressive disclosure at runtime: first the agent sees just a compact handle, then once it picks a skill it sees the detailed operators, and then if the contract is specified enough it just executes directly — only falling back to LLM reasoning for genuinely open-ended steps.
HOST_A: The numbers?
HOST_B: 57% token reduction at solve time. 2x faster. 57% cheaper. No accuracy regression. And — this is the part that maps onto PyranaHarness — when you compile with a strong model and serve to a smaller runtime model, the smaller model actually gets better.
HOST_A: Because the compiled contract is doing the reasoning work the smaller model would have struggled with.
HOST_B: Exactly. Steel-man: same affiliation risk on the benchmark, SkillsBench is from a closely overlapping group. And the harder objection — this assumes skill packages have classifiable structure. Real enterprise tool definitions are often semi-structured prose written by domain experts. If the "insufficient signal, raw fallback" path triggers often, the gains evaporate.
HOST_A: But our @tool() definitions already have signatures and docstrings. We have more structure than a worst-case enterprise.

HOST_A: Okay, last paper. PRISM from Yellow.ai.
HOST_B: This one is the silent killer problem. When LiteLLM rotates a model under us — same prompt, slightly different model behavior — our prompts can quietly start failing in ways that only surface as degraded downstream quality. PRISM is a continuous simulation and repair loop that catches this within 24 hours.
HOST_A: How does it work?
HOST_B: Five stages. First, given the requirements plus your tool schemas, an LLM generates a structured test suite — each test has a conversation script, objective pass criteria, and mock tool responses. Second, the current prompt is replayed turn-by-turn against the real platform architecture, with tool calls intercepted by mocks. Temperature fixed at zero for determinism. Third, a judge LLM evaluates each transcript on multiple dimensions — tool call presence, step ordering, routing, mandatory language. Fourth — and this is the clever part — when something fails, a surgical repair prompt modifies only the sections of the prompt responsible for the observed failure. It doesn't touch the parts that still pass. Fifth, it runs the whole suite nightly.
HOST_A: Numbers?
HOST_B: 99% production reliability across 35 enterprise agents over three weeks. Prompt authoring time dropped from a median of two days to under thirty minutes. Drift detected within 24 hours.
HOST_A: Objections?
HOST_B: Yellow.ai evaluating Yellow.ai on Yellow.ai's platform. No control condition — we don't know what the baseline reliability was without PRISM. And fixing temperature at zero for testing may understate real-world drift that happens at non-zero temperatures in production.
HOST_A: But the pattern is portable.
HOST_B: The pattern is very portable. We already have prompt version tracking and model rotation logs in LiteLLM. We already have pyrana-audit trails we can mine for mock tool response libraries. A nightly replay harness in pyrana-telemetry targeting our top-10 CxU extraction and agent system prompts — that's a one-sprint build. And it closes a gap we genuinely have.
